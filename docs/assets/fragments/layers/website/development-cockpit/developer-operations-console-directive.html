Developer Operations Console Directive

The Developer Operations Console is the active developer-facing operations layer for task controls, repository controls, workspace controls, environment controls, code review, runtime state, future terminal access, logs, and agent coordination.

The console now operates as a stabilized dockable system with Mini View and Full View modes, an independent content viewport, and a fixed summary footer.

The surface must remain minimal, calm, and market-ready while preserving modular runtime ownership beneath the interface.

Architectural Position

The Developer Operations Console belongs to the website development cockpit layer and coordinates developer-facing controls, repositories, workspaces, environments, runtime state, logs, agent interaction, and terminal access.

The homepage interaction panel shall remain the primary conversational input surface. Developer-specific operational controls shall live inside the Developer Operations Console.

The console shall not use overlays, compensating hacks, or unrelated owner files for layout correction. Each correction must be applied through the true structural owner.

Mode Model

  • Mini View: compact Developer Operations Panel inside the homepage stage stack with flattened horizontal interior alignment.
  • Full View: expanded developer console positioned below the Developer Mode topbar, portal-mounted outside the clipped stage composition, and aligned to the global platform rail formula.

The mode switch shall use one parent state token so all child zones inherit the correct layout behavior.

Full View shall preserve the Developer Mode topbar while suppressing only non-topbar viewport/body content.

Primary Console Zones

  • Search and command palette surface.
  • Tabbed operation domains.
  • Independent content viewport for tab content and operation results.
  • Fixed summary footer for project, repository, branch, and environment state.
  • Future terminal dock for Full View.
  • Future runtime logs and system state.
  • Future agent interaction and approval surfaces.

Tokenization Rule

The Developer Operations Console parent shall own shared layout and visual tokens for surface, border, radius, font size, control height, button styling, hover behavior, active state, gap, padding, panel width, Mini View, Full View, content viewport, and footer placement.

Child modules shall inherit from the parent and must not invent independent visual systems unless explicitly promoted into a governed token.

Topbar, tab, separator, profile/account, and icon behavior must reuse existing global systems where available. Local files may only apply ownership-specific structure and token consumption.

Terminal Placement

The terminal shall be introduced as a Full View bottom dock inside the Developer Operations Console.

The terminal shall remain structurally separate from the homepage interaction panel and shall support future runtime logs, command generation, scan output, and agent coordination.

The terminal shall not be implemented before the Full View shell, tab rendering, and parent-token styling system are stable.

Migration Rule

Developer-specific controls shall be migrated out of the legacy Developer Mode shell and into their correct Developer Operations Console tab domain.

  • Tasks: Create Prompt, Debug Issue, Fix Code.
  • Repositories: Repository selector, Connect GitHub, Discover repos, and repository runtime actions.
  • Workspaces: Project/workspace creation form and project creation runtime.
  • Environments: provider/agent list, provider activation runtime, and runtime scan action.
  • Code Review: Generate patch, Run review, Run tests, review artifacts, and locked review action runtime.
  • Archive: completed or historical developer operation records.

Legacy runtime handlers and center-column panels shall be removed from the Developer Mode shell only after their tab-owned replacements are implemented.

Developer Mode Topbar Rule

The Developer Mode topbar shall remain sparse and executive-level.

The approved topbar structure is Code, App, Docs, Notifications, Settings, and Account.

Code, App, and Docs are workspace switches. Notifications, Settings, and Account are utility controls requiring separate dropdown planning before implementation.

The topbar must not host project, repository, agent/provider, prompt, scan, debug, review, test, pull request, runtime, or log controls.

Account/Profile must reuse the global homepage profile/account control structure and must not invent local avatar, icon, circle, or state logic.

Left Sidebar Repurposing Rule

The Developer Mode left sidebar has been cleared of former workspace navigation items and shall remain available for future focused developer utilities.

Recommended future surfaces include Terminal, Logs, Command History, Approval Queue, Runtime Monitor, Local Bridge Status, and Agent Session Queue.

The left sidebar must not reintroduce broad operational navigation already owned by the Developer Operations Console tabs.

Full View Suppression Rule

When Developer Console Full View is active, all non-topbar homepage and Developer Mode surfaces shall be hidden. The Developer Mode topbar shall remain visible and untouched.

The shell-level Developer Mode selector must not be suppressed because it also contains the topbar. Suppression must target the body/viewport child layer only.

  • Keep: Developer Mode topbar.
  • Suppress: Developer Mode viewport/body content.
  • Suppress: homepage stage composition and interaction surfaces.
  • Do not suppress: shell root, topbar, header, navigation, or global menu owners.
